# 🚀 Expo × サーバー同居リポジトリの設計・構築ガイド

## 📌 はじめに（この記事の目的）

Expo（フロントエンド）とサーバー（バックエンド）を1つのGitリポジトリで管理する際、「どのようなフォルダ構成（アーキテクチャ）にすべきか」の決定版ガイドです。

数ヶ月後の自分が内容を忘れてしまっていても、このページを上から順に読むだけで、最適な構成の選択から初期構築までを迷わず再現できるように解説しています。

---

## 🧭 2つのパターンの比較（概要）

現場でよく使われる手法には、「パターンA：アプリ主役型」**と**「パターンB：プロジェクト全体一括型（モノレポ）」の2つがあります。

| 項目 | パターンA：アプリフォルダがルート | パターンB：ラッパーフォルダがルート |
| --- | --- | --- |
| **一般的な呼び方** | **シングルリポジトリ（単一構成）** | **モノレポ（Monorepo）構成** |
| **構造のイメージ** | アプリが主役。その中にサーバーが居候する。 | 独立したアプリとサーバーを、大きな箱で包む。 |
| **現場での採用率** | 約 30%（個人開発・スピード重視） | **約 70%（チーム開発・現場の主流・推奨）** |
| **後からの拡張性** | ❌ 別のアプリやWeb画面を足しにくい | ⭕️ フォルダを増やすだけで無限に拡張できる |

## 📄 各パターンの詳細・メリット・デメリット

### 📁 パターンA：アプリフォルダがルート（シングルリポジトリ）

### 🔹 フォルダ構成の見本

```
my-app/           ← 【重要】git init はここ（create-expo-app が自動作成）
├── src/          (Expoのソースコード)
├── server/       (サーバーのコードをここに入れる)
├── docs/         (仕様書やドキュメント)
├── package.json  (Expoのための設定ファイル)
└── app.json      (Expoの設定ファイル)
```

### 🔹 メリット

1. **初期構築が圧倒的に速い：** `npx create-expo-app` を実行して、その中にフォルダを作るだけなので1分で終わります。
2. **Expoの設定が簡単：** 全てがルート直下にあるため、VS Codeの拡張機能やパスの解決でトラブルが起きにくいです。

### 🔹 デメリット（後から辛くなる理由）

- **設定ファイルのごちゃ混ぜ（ルート汚染）：**
Gitの設定（`.gitignore`）や、コード整形（`ESLint` / `Prettier`）の設定が、Expoのものとサーバーのもので衝突しやすくなります。
- **CI/CD（自動化）の組みづらさ：**
GitHubにコードを上げたとき、「アプリだけ自動ビルドしたい」のに「サーバー側のコード変更」でもアプリのビルドが走ってしまうなど、細かい制御が難しくなります。

---

### 📁 パターンB：ラッパーフォルダがルート（モノレポ構成）★現場の主流

### 🔹 フォルダ構成の見本

```
my-project/       ← 【重要】git init はここ（手動で作成する全体の親）
├── app/          (Expoアプリ：ここにExpoの package.json や app.json が入る)
├── server/       (サーバー：ここにサーバー用の package.json やコードが入る)
└── docs/         (プロジェクト全体のドキュメント)
```

### 🔹 メリット（なぜ現場で選ばれるのか）

1. **関心の分離（安全性が高い）：**`app/` と `server/` が完全に独立しているため、サーバー側の設定を変更しても、アプリのビルドに絶対に悪影響を与えません。
2. **将来の拡張性が無限：**
サービスが成長して「管理画面（Web）」や「ランディングページ（LP）」を後から追加したくなっても、`my-project/` の直下に新しいフォルダを並べるだけで対応できます。
3. **CI/CD（自動化）が綺麗に組める：**
「`app/` 以下の変更ならアプリをビルド」「`server/` 以下の変更ならサーバーをデプロイ」という分岐が簡単に書けます。

### 🔹 デメリット

- **初期構築時のGitの競合：**`create-expo-app` コマンドは自動的に内部で `.git`（Git履歴）を作ってしまうため、手動でそれを消さないと「Gitの中にGitがある」というバグ（サブモジュール問題）の原因になります。

## 🛠 具体的な構築手順（コマンド集）

未来の自分が迷わないよう、ターミナル（MacのTerminalやWindowsのPowerShellなど）に打ち込むコマンドをそのまま記載します。

### 🛠 パターンAの構築手順

```bash
# 1. Expoアプリを自動作成（これで my-app フォルダと .git が作られます）
npx create-expo-app my-app

# 2. 作成されたフォルダに移動
cd my-app

# 3. サーバー用とドキュメント用のフォルダを作成
mkdir server
mkdir docs
```

---

### 🛠 パターンBの構築手順（注意点クリア版）

パターンBを作る際、一番のストレスは「Expoが勝手に作る不要な `.git` フォルダを手動で消す手間」**です。
ここでは、**「最初から不要な `.git` を作らせずに、一発で綺麗なモノレポを作るスマートな手順」を解説します。

```bash
# 1. プロジェクト全体の親となるフォルダを作成し、移動する
mkdir my-project
cd my-project

# 2. この親フォルダを Git のルートとして初期化する
git init

# 3. 【重要】公式のテンプレートを直接指定して、不要な .git を作らせずに Expo を導入する
# (末尾の「app」がフォルダ名になります。自由に変えてOKです)
npx create-expo-app app --template blank

# 4. サーバー用とドキュメント用のフォルダを横並びで作成
mkdir server
mkdir docs
```

### 💡 コマンド実行後の状態チェック

エディタ（VS Codeなど）で `my-project`（一番上の親）を開いてください。
以下のような見た目になっていれば**大成功**です。

- `my-project/` の直下に `.git` フォルダ（隠しフォルダ）が1つだけある。
- `app/` の中には `.git` が存在しない。

## 🚀 さらに一歩進むための未来へのヒント（Workspaces）

もしこのプロジェクトが「仕事」や「長期的な開発」になり、さらに本格的なモノレポにしたくなった場合は、ただフォルダを分けるだけでなく、パッケージマネージャーの「Workspaces（ワークスペース）機能」を導入することを検討してください。

### Workspacesを導入すると何が良いか？

1. **型（TypeScript）の共有：**`packages/shared` というフォルダを作り、そこにデータベースの型定義などを書いておくと、`app（フロント）` と `server（バック）` の両方から同じ型をインポートして使えるようになり、開発効率が跳ね上がります。
2. **コマンドの一元化：**
親フォルダ（ルート）にいるままで、`npm run start:app` や `npm run start:server` のように、両方の起動コマンドをコントロールできるようになります。

未来の自分が「もっと本格的にガチガチのモノレポにしたい！」と思ったら、次は「pnpm workspaces」**や**「Turborepo」というキーワードで検索してみてください。このページで構築した「パターンB」が、そのすべての土台になります。

```

├── README.md                       ← GitHub トップページに表示される概要
├── CLAUDE.md                       ← Claude Code 向けのプロジェクト指示書
│
├── my-app/                         ← Expo アプリ本体
│   ├── src/
│   │   ├── app/

│   │   │   ├── index.tsx           ← ホーム画面
│   │   │   └── explore.tsx         ← 通知デモ画面
│   │   ├── utils/

└── docs/                           ← ドキュメント・学習メモ
    ├── project_learning_journey.md ← 学習の流れ・全体解説
```

### .gitignore で除外しているもの

| 除外対象 | 理由 |
|---------|------|
| `my-app/android/` | `npx expo run:android` で自動生成されるネイティブコード |
| `my-app/ios/` | 同上（iOS ビルド時に生成） |
| `my-app/google-services.json` | Firebase の認証情報（公開禁止） |
| `node_modules/` | 依存パッケージ（`npm install` で再生成できる） |
| `.claude/settings.local.json` | Claude Code のマシン固有設定 |
